# openblock-qa

Тестовая оснастка для форка [OpenBlock Desktop](https://github.com/openblockcc/openblock-desktop) v2.6.3
под Arduino Uno и ESP32 (Windows x64).

Здесь лежит только то, что написано нами: скетчи, конфиги `arduino-cli`, проект симулятора
и тест-план. Сам OpenBlock сюда не входит — он клонируется из апстрима отдельно.

## Главный вывод: кириллица в пути ломает ESP32

Проверено компиляцией через `arduino-cli` из состава OpenBlock:

| Плата | Путь к тулчейну | Результат |
|---|---|---|
| Arduino Uno (AVR) | кириллица (`E:\уавцуа\OpenBlockDesktop\`) | компилируется |
| ESP32 | кириллица (`E:\уавцуа\OpenBlockDesktop\`) | **паника `Failed to get path name`** |
| ESP32 | ASCII (`E:\openblock-desktop\`) | компилируется |
| ESP32 | ASCII-тулчейн + кириллический build-путь | компилируется |

Ломается **только кириллица в пути к тулчейну**, то есть в папке установки. Путь сборки и имя
Windows-аккаунта на это не влияют. Падает Rust-утилита из состава esp32-тулчейна; AVR-тулчейн
кириллицу переживает.

Практическое следствие: установка per-machine в `C:\Program Files\...` даёт ASCII-путь всегда,
и ESP32 работает даже у пользователя с кириллицей в имени аккаунта. Установка per-user в
`C:\Users\Кириллица\...` или в кириллическую папку — ESP32 не соберётся. Чинится настройкой
инсталлятора, правки логики приложения не требуются.

## Структура

- `sketches/` — тестовые скетчи. `uno_rep` и `esp_rep` — представительные программы
  (analogRead + GPIO + Serial + PWM), а не только blink; `cyr_blink` — минимальный кейс
  для проверки кириллицы; `esp_basic`, `esp_pwm`, `ascii_blink` — узкие проверки.
- `configs/` — конфиги `arduino-cli`. `acli-cyr.yaml` указывает на кириллическую установку,
  `acli-esp32.yaml` — на ASCII. Пути в них абсолютные и заточены под конкретную машину —
  правьте `directories:` под себя.
- `wokwi-uno/` — проект симулятора Wokwi: Uno + светодиод на D13. В `fw/` лежит прошивка,
  собранная тем самым `arduino-cli` из OpenBlock, — симулятор гоняет ровно тот бинарь,
  который выдаёт приложение.
- `scripts/build-env.sh` — уводит npm/electron/gyp-кеши и TEMP с диска C: и ставит portable
  Node 16 первым в PATH. Пути прибиты к `E:/Загрузки/claude`.
- `docs/test-plan.html` — ручной тест-план на 13 кейсов: установка → первый запуск → выбор
  платы → COM-порт → сборка блоков → компиляция → прошивка → Serial → ESP32 → офлайн → драйвер.
- `scripts/repro-cyrillic.sh` — воспроизведение бага с кириллицей.

## Как воспроизвести баг

Нужна установка OpenBlock Desktop. Скрипт собирает один скетч дважды — из ASCII-пути и из
кириллического, — и показывает разницу:

```sh
sh scripts/repro-cyrillic.sh "E:/openblock-desktop" "E:/уавцуа/OpenBlockDesktop"
```

Первый аргумент — установка на ASCII-пути, второй — на кириллическом. Ожидаемо: первая сборка
проходит, вторая падает с `Failed to get path name`.

## Запуск симулятора

Требуется расширение `wokwi.wokwi-vscode` и бесплатная лицензия. Открыть папку `wokwi-uno/`
в VS Code, затем `Ctrl+Shift+P` → `Wokwi: Request a new License` (разово) → `Wokwi: Start Simulator`.
Плату выбирать не нужно — она описана в `diagram.json`.

Wokwi не создаёт COM-порт и не связан с OpenBlock: он только показывает поведение уже
скомпилированной прошивки без физической платы.
